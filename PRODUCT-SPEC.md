# VideoForge 产品规格文档（PRD）

> 三份文档的分工：`PRODUCT-PLAN.md` 讲商业与架构方向，`ARCHITECTURE.md` 讲代码现状与技术决策，**本文档讲"产品长什么样、每一页怎么做、按什么顺序实现"**。改产品先改这里，再动代码。
> 最后更新：2026-07-07

---

## 一、产品定位

**一句话**：把一篇文章/一个想法，变成一支带真人数字人头像、精确字幕、专业配音的讲解视频——用户只做选择题，不碰任何技术细节。

**目标用户**（按优先级）：
1. 自媒体创作者（抖音/视频号/B站），会用剪映但不会代码，想批量产出讲解类视频
2. 知识型个人 IP（想用自己的声音+形象出镜，但没时间真人录制）
3. 小团队内容岗（给公司号做产品讲解/行业解读）

**用户不需要知道的词**（这些词绝不允许出现在界面上）：workspace、stage、agent、scaffold、tsc、pipeline、job、HeyGem、乒乓、offset、cue。
**用户需要感知的只有**：作品、稿子、风格、配音、头像、字幕、进度、预览、导出。

**核心原则**：
- **选择题原则**——每个决策点给 2-4 个选项+推荐项，不给开放式表单（对齐 article2video skill 的"5 问决策"）
- **人话原则**——所有状态、报错都翻译成用户语言并附带"下一步该做什么"
- **费用透明原则**——每次要花钱（API token / 克隆 ¥10 / TTS）的动作，事前提示、事后显示消耗
- **能跑就先看**——任何时刻用户都能点开预览看当前成品，不用等全部完成

## 三、页面规范

### P0-1 首次使用引导（#/onboarding）

**触发**：`GET /api/settings` 返回的 llm 未配置或从未完成引导（settings 加 `onboarded: true` 标记）时，全屏接管。

三步（每步一屏，可跳过后两步）：
1. **连接大脑**：两张大卡片二选一——「我有 Claude Code 订阅（本机已登录，推荐）」/「我用 API key」（选后展开供应商+key 输入+测试按钮，通过才能下一步）
2. **连接声音**：MiniMax key 输入 + 测试；下方小字"没有 key？跳过，稍后在设置里补"
3. **准备形象（可选）**：介绍数字人功能 → 「稍后再说」/「去素材库上传」

完成后写 `onboarded: true`，进工作台。**验收**：全新 settings.local.json 启动 → 必然先看到引导；配好订阅模式 → 测试按钮变绿 → 能进工作台。

### P0-2 新建作品向导（#/new）

五步向导，顶部步骤条，每步一屏。**这五步与 article2video skill 的 5 个决策一一对应**——产品即规范。

**Step 1 · 内容来源**（四选一 tab）：
- 🔍 帮我找选题：方向关键词输入 + 「开始搜索」→ 结果卡片列表（标题/来源/两句话推荐理由）→ 选一条
- 🎵 抖音链接：粘贴分享文本 → 提取 → 展示提取到的文案预览 → 确认使用
- 提取完成后创建作品必须停在“原文确认”；原文按自然句段排版，可手动编辑和保存，只有用户点击确认按钮后才开始生成口播稿。
- 🔗 文章链接：URL → 抓取正文预览 → 确认
- ✍️ 直接贴文字：大文本框
- RSS 订阅从主流程移除，收进设置页"高级"折叠区（保留 API）

**Step 2 · 确认稿子**：点「生成口播稿」→ 进度动画（"AI 正在读文章、写口播稿，约 3-5 分钟"）→ 完成后左右对照展示（左原文/右口播稿，手机上上下）→ 操作：「就用这版」「换个写法」（重新生成）「我来改」（行内编辑）。outline 不展示给用户（技术产物），只在"高级"折叠里可见。

**Step 3 · 选风格**：主题画廊，每个主题一张真实缩略图（一次性批量渲染 23 张存 dashboard/public/themes/）。按内容基调置顶推荐 2 个（技术→midnight-press、新闻→newsroom，规则来自 article2video DEFAULTS），其余折叠在"更多风格"。
- “文字与排版”与“字幕预设”必须在选项右侧显示主题色块预览：字号与排版密度切换时同步改变标题比例、内容数量和留白；字幕样式、位置或开关切换时同步改变字幕条、字幕安全带和所在区域。窄屏时预览移到选项下方，不得横向溢出。

**Step 4 · 配音**：默认卡片直接展示当前音色+已调教参数（语速 1.12/情绪 angry）+「🔊 试听」；可切换情绪预设（激昂/平静/欢快 → 映射 angry/calm/happy）；「不要配音」小开关（纯网页交付）。

**Step 5 · 数字人头像**：三选一——「用我的形象」（素材库选择器）/「暂不出镜」/「先做视频，头像以后加」。选形象时展示右侧竖屏窗口示意图（360×640 右侧布局，来自 DEFAULTS）。

底部「开始制作」→ 创建 job → 跳转作品详情。**验收**：全程无一个技术词；每步可后退；刷新不丢已选项（草稿存 localStorage）。

### P0-3 工作台（#/works，默认页）

作品卡片流（非表格）：封面缩略图（有预览后截第一屏，之前用主题色块）+ 标题 + 状态徽章 + 一句话状态 + 主操作按钮。

状态徽章用户语言映射（**全站统一，写进 dashboard/src/lib/statusText.ts，禁止散落硬编码**）：

| 内部状态/stage | 用户看到 | 主操作按钮 |
|---|---|---|
| queued | 排队中 | — |
| running + script_outline | ✍️ 正在写口播稿（约3-5分钟） | — |
| waiting_approval + gate_script | 📝 稿子写好了，等你确认 | 「去看稿子」 |
| running + scaffold | 🏗️ 正在搭建画面框架 | — |
| running + chapter_gen | 🎨 正在逐章制作画面（约20-40分钟） | — |
| waiting_approval + gate_chapters | 👀 画面做好了，等你验收 | 「去预览」 |
| running + audio_synth | 🎙️ 正在合成配音 | — |
| running + subtitle_cues | 💬 正在生成精确字幕 | — |
| waiting_approval + done / render | ✅ 制作完成 | 「去导出」 |
| failed | ⚠️ 这一步出了点问题 | 「重试」+「查看原因(人话)」 |

**验收**：任何状态的作品卡一眼能看懂"现在在干嘛、要不要我做什么"。

### P0-4 作品详情（#/works/:id）
桌面工作区宽度上限 1560px，主面板与对话栏随可用视口高度扩展；高屏优先扩大正文/稿件阅读区，900px 以下回归自然高度的单栏布局，禁止横向溢出。
桌面端主面板与对话栏必须等高，超长内容只在各自区域内滚动。原文确认步骤不显示对话栏，原文支持阅读排版、手动编辑和保存；用户点击确认前不得自动进入下一步。对话输入框使用 Enter 发送、Shift+Enter 换行。

> 2026-07-14 已切换为静态预览：工作台 iframe、整片播放和服务端渲染统一访问 `/preview/:id/`。用户点击“加载/刷新预览”时执行懒构建与脏检查，不再感知或启动 per-job dev server；构建失败必须在人工作品验收前直接显示可操作错误。画面、配音字幕或数字人的对话修改完成后，后端必须先重建预览，工作台再自动刷新左侧 iframe，并保持在当前验收章节。
> 逐章画面生成期间采用增量预览：每当有新章节完成并通过检查，自动重建当前静态预览；已完成章节立即可切换查看，正在生成或排队章节保持禁用。生成服务中断时仍保留已完成章节的预览能力，不得要求全部章节重新完成后才能查看。
> 章节列表与进度行只显示面向用户的中文章节标题，优先读取 `presentation/src/registry/chapters.ts` 的 `title`；目录名和模型工具动作（如 `03-imitation`、`read_file`）属于内部标识，禁止直接显示在界面上。

从上到下四区：
1. **进度区**：横向步骤条（人话阶段名：写稿→确认→搭建→画面→验收→配音→字幕→完成），当前步高亮+预计耗时；失败步红点+「重试」
2. **行动区**（仅需要用户时出现）：稿件确认（稿子全文+「通过」「换个写法」）或章节验收（大预览+反馈框）
3. **预览区**：16:9 iframe（自动起 dev server，用户无感知——按钮文案「打开预览」而非"启动预览服务"）+「新窗口全屏」（录制用，自动带 ?auto=1）
4. **修改对话区**：聊天式（替代现在的单条反馈框）——消息历史 + 输入框 +「只改第几章」下拉（章节名而非文件夹名）；每条消息显示处理状态，影响画面的修改完成后自动重建并刷新左侧预览，不要求用户手动点击刷新。
   - 口播稿修改时左侧显示处理中遮罩；完成后必须重新读取 `script.md` 并明确提示稿件已更新。
   - 逐页生成支持“全局修改 / 当前第 N 章”两种作用域；处理中遮罩旧预览，完成后自动刷新并明确提示修改范围。
   - 配音与字幕生成必须显示持续 loading、当前阶段和可用百分比，不能只显示“正在校验”。
   - 启用字幕时，章节画面按字幕位置预留无正文安全带：底部 190px、下三分之一 290px、顶部 170px；数字人与字幕同时启用时取两者安全区并集。

**导出区**（完成后替代行动区）：录制三步图文指引（人话版 render note）+「复制录制地址」按钮；M3 换成「一键导出 mp4」。

**重新操作提示**：任何“重试 / 重新生成 / 重新导出”按钮旁必须同时说明三件事：从哪个用户可理解的环节开始、会重做哪些内容、会保留哪些内容。重新导出只重录画面并混音，复用 PPT、配音、字幕和数字人；更换数字人只重做口型与画面接入，完成后重新验收数字人并导出，不返回 PPT 环节。覆盖已有成片前需要二次确认。
**隐藏**：workspace 路径、原始日志（收进右上角"…"菜单→"技术详情"，开发者自己用）。

### P1-5 素材库（#/assets）

两个 tab：
- **我的声音**：音色卡片（名称/试听/是否默认）+「＋克隆新声音」向导（三步：录音要点说明→上传→费用确认 ¥10→克隆→自动试听）。现设置页克隆卡片迁移至此。
- **我的形象**：出镜视频卡片（缩略图/时长/是否默认）+「＋上传形象」（上传 → 服务端存 workspaces/_assets/avatars/ → 提示"10-60 秒正面出镜、光线均匀"）。为 avatar stage 和远程 HeyGem 的 avatar_key 复用做准备。

### P1-6 设置（#/settings）——瘦身

保留五卡片但重组：「模型」「声音」（key+参数，克隆移到素材库）「数字人服务」「内容源」+ 新增「高级」折叠区（RSS 管理、技术日志开关、workspace 根目录展示）。每卡片右上角状态点（绿=已配置可用/灰=未配置）。

## 四、文案与错误规范

- 报错三段式：**出了什么事（人话）+ 可能原因 + 现在怎么办（按钮）**。例：TikHub 401 → "抖音解析服务拒绝了请求 · 可能是 key 失效或额度用完 · [去设置检查 key]"。原始错误收进"技术详情"折叠。
- 耗时动作必须有：进行中动画 + 预计时长 + 完成通知（页面内 toast；浏览器 Notification 作为 P2）。
- 费用提示统一组件：`<CostHint>`（克隆 ¥10 / 全量配音约 ¥X / API 模式显示 token 消耗）。
- 语气：说人话、短句、不卖萌不堆 emoji（状态徽章里的单个 emoji 除外）。

## 六、成功标准

- 新用户（无技术背景）从打开产品到拿到第一支可录制视频 ≤ 40 分钟，中途求助 ≤ 1 次
- 界面零技术词（用 grep 验收：workspace/stage/agent/job 不出现在任何用户可见文案里）
- 每个等待状态都有预计时长；每个失败都有一键重试 + 人话原因
- 换风格重新生成时不得继续展示旧版画面；生成中明确遮罩，全部章节生成并检查完成、进入画面验收后自动加载新版预览。
- 章节进度中文不得乱码；新任务统一写 UTF-8，并兼容读取 Windows 中文代码页产生的历史或在途进度文件。
