# VideoForge 产品化规划（面向普通用户版）

> 2026-07-07 起草。v1（个人工具形态，见 README.md）已验证流水线跑通；本文规划把它推给**没有 Claude 订阅、没有本地 GPU、不懂命令行**的普通用户需要补齐的东西。

---

## 一、总体架构：三层拆分

```
┌─ 用户侧（浏览器 / 桌面壳）───────────────────────────────┐
│  控制台 UI：选题 → 文案 → 风格 → 生成 → 调试 → 预览 → 导出   │
│  本地保存：用户自己的 LLM key / MiniMax key / voice_id      │
└──────────────┬───────────────────────────────────────┘
               │ HTTPS
┌─ 生成服务（可跑在用户本机，也可跑在你的服务器）────────────┐
│  ① 模型接入层（Provider Adapter）— 本文 §二               │
│  ② 流水线引擎（现有 videoforge server，阶段幂等+重试）      │
│  ③ 工件存储（workspace 文件系统 + SQLite）                 │
└──────────────┬───────────────────────────────────────┘
               │ 任务队列（HTTP + 轮询，与现有 HeyGem API 同构）
┌─ GPU 网关（你的电脑 / 未来的 GPU 服务器）─────────────────┐
│  HeyGem 对口型服务（7861，已有 frp 隧道 heygem.yyagent.top）│
└──────────────────────────────────────────────────────┘
```

关键决策：**生成服务和 GPU 网关分离**。普通用户电脑性能不重要——重活只有两类：LLM 调用（走他们自己的 API key，云端）和对口型（走你的 GPU 网关，远程）。用户本机只需要能跑 Node + 浏览器。

## 二、模型接入层（BYO API Key）

现状：v1 用 `claude -p` 走你的订阅——**不可分发**（订阅不能共享，且用户机器上没有登录态）。

方案：抽一个 Provider Adapter，生成环节改为直接调 LLM API。两种模式并存：

| 模式 | 适用 | 实现 |
|---|---|---|
| **订阅模式**（保留） | 你自己 | 现有 `claude -p` 路径不动 |
| **API 模式**（新增） | 分发给用户 | Claude Agent SDK（headless，`ANTHROPIC_API_KEY`）为首选——skill 文件原样可用；同时留 OpenAI-compatible 适配（DeepSeek/Kimi/GLM/通义……国内用户拿得到的 key），走"把 SKILL.md 内容注入 system prompt + 工具循环"的降级实现 |

- 设置页一个「模型供应商」卡片：下拉选供应商 → 粘贴 key → 「测试连接」按钮（发一个 1-token 请求验证）→ key **只存用户本地**（桌面壳存 OS keychain / Web 版存 localStorage 加密），**你的服务器永远不落库、不经手**。
- 章节生成质量强依赖模型能力，UI 上标注推荐档位（如 claude-sonnet-5 起步），弱模型给出预期提示。
- 计费透明：每个 Job 结束显示本次消耗的 token 数（各 SDK 都返回 usage），让用户对自己的 key 花费有感知。

## 三、可视化流程（逐环节回答你列的问题）

### 1. 选题发现
v1 已有 RSS 抓取。补充：热榜聚合（微博/知乎/36kr/HN 按赛道可配）、关键词订阅、手动贴 URL/全文三入口。每条候选带"适合做视频吗"的一句话 AI 评估（标题党程度/信息密度/时效性）。

### 2. 口播文案生成 + 审批
v1 已有（script_outline 阶段 + 审批门）。补充：文案编辑器里**行内改写**（选中一段 → "改口语点/缩短/换例子"），比整篇重生成省 token。

### 3. 风格选择
主题画廊：23 个主题各配一张缩略截图（可以从 skill 的 themes/ 批量渲染一次性生成），点选即定。默认按内容基调预选（技术→midnight-press，新闻→newsroom——规则已在 article2video skill 里）。

### 4. 并行生成（"多线生成能不能实现"——能）
章节天然独立（每章一个组件文件夹），并行毫无障碍。v1 串行只是订阅额度限制。API 模式下：
- 并发度 = 用户可配（默认 3），每章一个独立 agent 任务；
- UI 上每章一张卡片，实时状态（排队/生成中/自检中/完成/失败），失败单章重试；
- 这正是已验证过的模式（harnessH5video 就是 6 个 subagent 并行建章）。

### 5. 对话式调试（"跟 API 对话调试哪里生成得不好"）
每章卡片带一个聊天框 → 用户的话 + 该章代码上下文发给一个 scoped agent → 改完 HMR 自动刷新预览。v1 的"章节验收门反馈框"就是这个的雏形，升级为多轮对话（保留会话历史，支持"再改回去"）。全局也放一个聊天框处理跨章问题（"所有标题都大一号"）。

### 6. PPT/成片展示（"单开窗口还是如何展示"）
**推荐：控制台内嵌 iframe 预览 + 「全屏/新窗口」按钮双轨。**
- 内嵌 iframe（16:9 等比缩放）：调试期用——反馈聊天框就在旁边，改完即刷，不用切窗口；
- 「新窗口打开」按钮（弹独立窗口 `?auto=1`）：验收和录制用——录屏必须无 UI 干扰全屏。
- 这不是二选一，两个按钮都放。技术上 dev server 已在跑，iframe 直接指 `localhost:<port>`，零额外成本。

### 7. 音频合成
- 设置页「MiniMax」卡片：粘贴 API key（同样只存本地）+ 试听按钮。
- 语速/情绪不给滑块海洋，给**预设**：默认直接用调教好的 1.12/angry（按内容基调映射情绪），高级折叠面板里才放微调；改动后走"单句 A/B 试听 → 确认 → 全量合成"流程（这套已验证，成本几分钱）。

### 8. 声音克隆上传（"克隆的声音到底该如何上传"）
向导式三步，重点是**防重复克隆**（克隆按次收费 ~10 元，合成才便宜）：
1. 上传 20-60s 清晰人声样本（自动转 ASCII 文件名——OSS 中文名的坑已踩过）→ 前端提示录音要点；
2. 明示"克隆是一次性付费动作（约 ¥10，从你的 MiniMax 账户扣）"→ 确认 → 调 `/v1/voice_clone` → 得到 voice_id；
3. voice_id 存入用户本地 profile，之后所有项目直接复用；界面上只有「更换声音」才会再触发克隆。

### 9. 对口型（远程 HeyGem，"性能低的电脑如何走我的服务"）
你已有全部零件：HeyGem server 本身就带 `avatar_key` 形象库、`user_id` 目录隔离、任务队列和 OSS 直传；frp 隧道（heygem.yyagent.top）已在跑。要补的是**网关层**：
- **鉴权**：给每个用户发 token（简单起步：你手工发；后续接支付/额度）；
- **排队与公平**：GPU 单卡串行，队列长度和预计等待时间要在 UI 上实时显示（"前面还有 3 个任务，约 6 分钟"）；单任务限时（现有 600s 超时够用）；
- **形象库复用**：用户第一次上传出镜视频时 `save_as_avatar=true` 存成 avatar_key，之后只传音频不再传几十 MB 的视频——省带宽也省你的磁盘；
- **产物回传**：生成完控制台自动下载 `video/file/{id}` 进 workspace，走后续的交叉相关对齐流水线（脚本已在 skill 里，服务端跑，用户无感知）；
- **风险边界**：你的家用电脑当生产服务器（断电/占用/IP 变动），中期迁到带 GPU 的云主机或二手矿卡盒子，网关协议不变。

### 10. 导出
v1 是录屏指引（最可靠）。v2 上服务端自动成片：Playwright 无头开 `?auto=1` + ffmpeg 录制合成 mp3 音轨 → 直接吐 mp4。这步做完，用户全程零手工。

## 四、分阶段路线图

| 阶段 | 内容 | 判断标准 |
|---|---|---|
| **M0**（已完成） | v1 个人工具：订阅模式全流水线 | 自己能日常出片 |
| **M1** | Provider Adapter（API 模式）+ MiniMax key/克隆向导 + 头像 stage 进流水线（含远程 HeyGem 网关最小版：token+队列） | 一个不懂命令行的朋友在自己电脑上装好并出一支片 |
| **M2** | 并行章节生成 + 多轮调试对话 + 主题画廊 + 用量/费用面板 | 3-5 个种子用户稳定使用，收集付费意愿 |
| **M3** | 服务端自动成片（Playwright+ffmpeg）+ 分发形态定型（Electron 桌面壳 vs Web SaaS）+ 支付/额度 | 可对外售卖 |

## 五、待定问题（需要你拍板，不影响 M1 动工）

1. **分发形态**：Electron 桌面壳（你有 XiaMuDesktop 打包经验，key 存 keychain 更安全）vs 纯 Web + 本地 server（安装门槛更低）——建议 M1 先 Web，M3 再定；
2. **商业模式**：卖软件（一次性整合包，你有先例）vs 订阅 + GPU 算力抽成 vs 免费引流带 HeyGem 算力收费；
3. **合规**：声音克隆和数字人有实名/授权合规要求（各平台对 AI 生成内容标注也有规定），对外前需要一版用户协议；
4. **多模型下限**：非 Claude 系模型做章节代码生成的质量地板在哪，M1 期间用 DeepSeek/GLM 各跑一支片实测，决定是否"只推荐 Claude API"。

## 六、B 形态分发路线图（当前正式执行顺序）

### 目标

达到“把一个安装包发给朋友，他填自己的密钥就能独立出片”的标准：

`下载/解压 → 首次配置 → 粘贴文章 → 等待成片`

本路线只覆盖单机可分发的 B 形态，不提前引入账号体系、额度计费、多租户隔离或云端中继。

### 第一梯队：产品先站稳（打包前）

#### B1. 数字人阶段拆分与确定性接线（✅ 2026-07-15 已实施：avatar_media/avatar_wire + registry 契约；旧作品 LLM 回退）

- 将 `avatar_gen` 拆为 `avatar_media` 与 `avatar_wire`：
  - `avatar_media`：HeyGem 推理、按产物哈希 checkpoint、失败重试不重复推理；
  - `avatar_wire`：把数字人产物接入视频流水线。
- `Subtitle`、`AvatarPresenter` 组件及 CSS 改为仓库内确定性模板，禁止让 LLM 临时生成这类基础接线代码。
- 验收：同一输入可重复得到相同阶段产物；接线失败可单独重试；缺失 CSS/import 不再破坏布局；一次失败不会重新跑 HeyGem 推理。

#### B2. 数据目录与代码分离（✅ 2026-07-15 已实施：dataRoot 四级解析 + 冷启动供给）

- 将 `data.db`、`workspaces/`、`settings.local.json` 等运行时数据迁移到 `%APPDATA%\\VideoForge\\`。
- 所有运行时路径统一从 `dataRoot` 配置解析，仓库/安装目录只读代码和静态资源。
- 验收：安装目录无写权限时仍可启动、创建作品、渲染和升级；升级不会覆盖用户数据；旧目录可执行一次性迁移。

### 第二梯队：外部能力降级开关

#### B3. 声音音色（2026-07-15 用户拍板：走克隆自费流程，不做系统音色兜底）

用户用自己的 MiniMax key 走产品内已有的声音克隆向导（克隆费用由用户自己的 MiniMax 账号承担，约 9.9 元/次）；GongheJiucun02 是作者个人音色，分发版不预设。计费/代付细节后置到 C 形态。原文案：

- 默认使用 MiniMax 系统音色，不依赖个人克隆音色 `GongheJiucun02`。
- 设置页提供音色选择器；现有克隆向导作为进阶功能保留。
- 验收：新用户只填 MiniMax key 即可合成语音；克隆音色不可用时有明确降级提示。

#### B4. 数字人策略与远程服务

- 默认关闭数字人，避免把 GPU 作为基础依赖。
- 进阶选项支持远程 HeyGem 网关（`heygem.yyagent.top`），保留 token 配置字段。
- 修复 HeyGem 与 frpc 的机器自启和常驻，加入服务健康检查。
- 验收：无 GPU 时默认流程仍能完成；开启远程数字人时能看到连通性、排队和失败原因。

#### B5. 抖音提取与 ASR 降级（✅ 2026-07-15：TikHub 未配置时抖音入口自动禁用；ASR 可在设置页检查连通性，缺失时明确提示仅可处理带完整字幕的作品）

- TikHub key 未配置时隐藏抖音提取入口，并显示配置位置。
- Whisper/远程 ASR 未配置时禁用相关入口，不阻塞普通文章转视频流程。
- 验收：缺少可选 key 不出现未处理的 401/隧道错误。

### 第三梯队：打包与首次体验

#### B6. 自包含分发包（✅ 2026-07-15 首版：scripts/package-portable.ps1 产出便携目录，Node、共享依赖、ffmpeg/ffprobe 均随包并自动供给；冷启动实测通过。待：压缩安装器）

- 打包 Node runtime、server、dashboard 及 ffmpeg/ffprobe（约 100 MB）。
- 第一版采用绿色便携包：解压后双击启动脚本；后续再评估 Electron/Tauri 外壳。
- 浏览器复用系统 Chrome/Edge；未检测到可用浏览器时给出人话提示和安装指引。
- 验收：干净 Windows 机器无需开发环境即可启动并完成一次渲染。

#### B7. 首次启动向导（✅ 2026-07-15：三步全屏向导已落地，模型必须验证，配音与形象可跳过，完成后直接进入第一支作品创建）

- 引导填写 LLM key 与 MiniMax key。
- 逐项调用现有测试接口验证连通性，失败时说明下一步。
- 引导用户完成第一支片，默认路径不要求理解技术概念。
- 验收：新用户从下载到第一支成片不需要打开命令行或手动编辑配置文件。

#### B8. 收尾能力（✅ 2026-07-15：统一错误翻译、脱敏诊断导出、GitHub Release 更新提示已落地）

- 错误信息统一人话化，并提供可执行的下一步建议。
- 增加“一键导出日志”，便于远程排障。
- 增加 GitHub Release 更新检查，仅提示，不自动覆盖用户数据。

### 第一张工单（执行批次）

按依赖顺序并行推进：

1. 静态化：把 Subtitle/AvatarPresenter 及 CSS 收入仓库模板；
2. 数据目录分离：引入 `dataRoot` 并完成旧目录迁移；
3. avatar 拆分：先落 `avatar_media` checkpoint，再接 `avatar_wire`。

三项完成后的共同验收：在干净数据目录中连续完成两次渲染；第二次命中媒体 checkpoint；安装目录保持只读可运行；数字人关闭时普通流程不受影响。

### 不在 B 形态范围内

账号体系、额度计费、多租户隔离、云端中继和完整商业化支付不纳入本阶段，待验证分发闭环后另行评估。

## 七、C 形态路线图（成熟软件：账号 + 远程数字人 + 激活闲置云资产）

### 2026-07-16 现状对账与立即行动

**基础设施事实核对**（用户印象 vs 实际）：域名 yyagent.top **已经绑定并在服务**（DNS → 阿里云上海 106.14.151.37，nginx 已配 heygem/asr/videomix 子域 → frps 隧道，2027-01-23 到期）。真正缺的不是"绑域名"，而是下面三件：

前提：§六 B 形态通过"朋友拿安装包能独立出片"的验收。C 形态回答"做成成熟软件"：**别人不需要 GPU，数字人调用你本机的 HeyGem**；账号、短信登录、额度记账全部复用你已有但闲置的资产。

### C1. 算力面可靠化（把"你家电脑"变成"服务"）

- 服务自启：schtasks 注册 startup.ps1（MACHINE-INDEX §四已有现成修法）；frpc 常驻 + 断线重连。
- HeyGem 服务端加 token 鉴权（客户端 `heygem.token` 字段已预留，server_v2 侧补校验）。
- 掉线告警：云端 cron 探活隧道，失败**发短信给你**（阿里云 SMS 复用，一条告警通道零成本）。
- **排队与配额是硬约束**：单 GPU 串行，按 60 秒视频约 5 分钟推理估算，理想日吞吐 ~200 段。中继层必须有任务队列（排队位置对用户可见）+ 每用户每日额度。这决定初期规模上限：**10-30 个活跃用户**，超出即需第二张卡或降级策略——这是物理边界，规划期就承认它。

### C2. 云控制面薄中继（yyagent.top 新增 `/api/relay/*`）

- `POST /api/relay/heygem/submit`：校验登录 token → 扣额度 → 入队 → 转发隧道；`GET /api/relay/heygem/task/:id` 轮询转发。ASR 同构一组。
- 上传大小限制、审计写 `usage_logs`。
- 客户端只改配置：`heygem.baseUrl` 从 `127.0.0.1:7861` 换成中继地址 + 登录 token——设置页字段全部现成，代码零新增。

### C3. 客户端账号化

- 登录页（短信验证码，走现成 `/api/auth`），token 存本地数据目录。
- 一次登录解锁：远程数字人 + 云端抖音提取 +（后续）代付模型。
- **未登录 = 完整 B 形态**（BYO key + 无数字人/本地数字人）——账号是增值项不是门槛，降低传播阻力。

### C4. 计费（先记账，后收费）

1. 阶段一：只记账不收费（`usage_logs` 记每次 relay 调用），内测期观察真实成本结构。
2. 阶段二：每日免费额度 + 简单充值（先手工开通，验证付费意愿后再接支付）。
3. LLM/TTS 代付放最后（资金风险最大）；BYO key 长期保留为兜底。

### C5. 安全与合规底线

- 数字人素材是真人肖像：仅允许上传本人素材 + 一页纸使用条款 + 服务端留提交记录。
- 用户内容会流经你的 GPU 机器：隐私说明写清"处理后保留多久、是否删除"；HeyGem 工作目录与你个人数据隔离。
- relay 层限流防滥用；MariaDB mysqldump 定时备份（MACHINE-INDEX 已列待办）。

### C6. 运营件

- 分发页（GitHub Releases 或 yyagent.top 静态页）+ 客户端启动时查 `/api/version` 提示更新。
- 反馈渠道（微信群）+ 客户端日志一键导出。
- 对内测用户明说算力现实：**算力面在你家里，停电/关机=数字人服务中断**，公布"服务时段"比假装高可用更成熟。

## 八、表现力路线（镜头语言 + 情绪化朗读，2026-07-16 规划）

目标：向优秀科普博主的观感看齐——镜头推拉、局部特写、灵动强调、时缓时急的朗读。
架构优势：成片 = 真实网页播放实录，**任何网页动画都是免费的视频效果**，渲染管线零改动。

### 效果 v1 ✅（2026-07-16 落地）

- **CSS 摄像机层**（CameraLayer）：focus 推近 / pan 平移 / spotlight 聚光 / overview 拉远，
  registry/cameraCues.ts 数据声明，字幕与数字人在层外不受影响。
- **逐词跟读高亮**（WordMark）：口播念到某词的瞬间画面元素同步点亮——复用字幕 cue 时钟，
  独门优势（博主要手动 K 帧，我们自动）。粒度 cue 级（1-2.5s），v2 升级 words.json 字级。
- **数字滚动**（Counter）+ **手绘圈注**（Annotate circle/underline）。
- **四层校验**：受限原语（AI 只能选件传参）→ cameraCheck 静态校验（词表/倍数/每章 ≤3 预算，
  违规即章节生成失败）→ 运行时 target 未命中记 window.__vfCameraMisses 原地不动 →
  人工门 gate_chapters 预览实看。纪律入 CHAPTER-CRAFT「镜头与效果件」节。

### 效果 v2（v1 经真实作品验收后）
