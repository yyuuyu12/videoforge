# 讲师头像 + 精确字幕（web-video-presentation 的配套 Skill）

这个 Skill **不是**独立流程，是 `web-video-presentation` 做完之后的下一段：那个 Skill 负责"文章 → 网页演示 + 口播音频"，这个 Skill 负责"网页演示 → 看起来像真人讲解的短视频"。两者一起用，顺序固定：**先跑完 web-video-presentation 的 Phase 1-3（口播稿/outline/网页开发/音频合成都完成），再开始本 Skill**。

## 什么时候用这个 Skill

- 已经有一个 web-video-presentation 项目（`presentation/` 目录、`public/audio/` 下已有合成好的口播音频）
- 用户想要：真人头像小窗口、更精致的字幕、或抱怨过"手机上字看不清"
- 准备发到短视频平台之前的最后一轮打磨

如果用户只是说"把文章做成视频"，先用 `web-video-presentation`；只有当那部分完成、且用户提到头像/字幕/手机观感这几件事之一时，才进入本 Skill。

---

## 工作流总览

```
前置条件：web-video-presentation 的 script.md / outline.md / presentation/ 已完成，
         public/audio/<chapter>/<step>.mp3 已存在。

Phase S   精确字幕
   S.1  用 TTS 的 subtitle_enable 重新合成音频，拿到逐字时间戳
   S.2  把时间戳分组成"~10 字一屏"的字幕分段（规则见 references/SUBTITLE-SYNC.md）
   S.3  接入播放器：字幕跟音频真实播放位置同步切换，不是估算
   ▼
Phase A   讲师头像（可选 —— 需要用户提供一段本人出镜视频，且要跑对口型模型，耗时耗资源）
   A.1  问用户要驱动视频 + 确认要用的对口型模型/API + 确认布局（默认右侧竖屏大窗口）
   A.2  乒乓循环拉长驱动视频，按 ≈10s 分段连续切片（游标只进不退，且跳避乒乓翻转点）
   A.3  逐段调用对口型模型生成 → 拼接成一条完整视频
   A.4  跟模型输出里自带的音频做交叉相关求每步真实 offset（不是按时长比例估算）
   A.5  运行时把视频"从动"到音频时钟（每帧 offset+audio.currentTime，不是每步 seek 也不是自由播放）
   ▼
Phase M   手机字号体检
   M.1  过一遍 references/MOBILE-SIZING.md 的检查清单，重新排版而非简单加大字号
   M.2  跑 scripts/subtitle-overlap-sweep.js，确认没有内容被字幕条压住
   ▼
交付：full HD 网页可录屏，或已有对口型合成视频可直接剪辑成片
```

**Phase S 必须做**；**Phase A 只有用户主动要求头像小窗口时才做**（涉及真实视频素材和外部模型调用，不要在用户没提供素材时假设）；**Phase M 在前两个 Phase 做完后自动跑一遍**，不需要用户额外确认——这是纯粹的质量检查，没有破坏性。

不需要在每个子步骤都停下来问用户——按 Phase S → (A) → M 的顺序做完，跑不通或需要用户提供素材/密钥时再停下来问。

## Phase 0 · 开始前，先自查 web-video-presentation scaffold 的一个已知 bug

这条跟头像/字幕本身无关，但**每一个用 web-video-presentation 新建的项目都要检查一遍**，所以放在本 Skill 里（这样不依赖某次对话的记忆，任何新项目、任何新会话都能看到）：

`presentation/src/hooks/useAutoMode.ts` 和 `useStepper.ts` 各自独立监听全局 `Space` 键——一个用来"解锁自动播放"，一个用来"手动前进一步"。两个监听器会在同一次按键里都触发，导致用户按 Space 启动自动播放（页面上正是这样提示的）的同一瞬间，也被当成"手动前进了一步"，第 0 步的画面/音频会被跳过，正式录制会丢开头一句话。

检查方法：打开 `presentation/src/hooks/useStepper.ts`，看 `useStepper` 有没有接受一个 `suppressSpace` 选项（通过 ref 实现，不触发重新订阅），且 `App.tsx` 里有没有传入 `suppressSpace: mode === "auto" && !autoStarted`。没有就补上——具体实现抄本 Skill 维护者本地任意一个已完成项目的 `useStepper.ts`/`App.tsx` 即可。这是本地补丁，没有上游到 garden-skills 仓库，所以每个新项目都要自己补一次。

## Phase S · 精确字幕

### S.1 重新合成音频，要真实的逐字时间戳

字数估算的字幕（"一个字 250ms"之类）短句还凑合，长句会明显滞后。只要 TTS 供应商支持返回时间戳就应该用，比事后猜准得多。

MiniMax `t2a_v2`（本项目验证过的参考实现）：请求体加两个字段

```json
{ "subtitle_enable": true, "subtitle_type": "word" }
```

返回体 `data.subtitle_file` 是一个 24 小时有效的下载链接，内容是 JSON 格式的逐字/逐句时间戳（毫秒）。`subtitle_type` 可选 `sentence` / `word` / `word_streaming`；要做"每次几个字切换"的字幕用 `word`。**这个字段第三方文档不一定写全，实际字段名以下载下来的 JSON 为准**——先对一两条真实数据探个路，再写解析代码，不要凭文档假设字段名直接批量跑。

换了别的 TTS 供应商：先确认它是否支持时间戳（多数主流 TTS 都支持，只是常常要显式传参开启），没有的话就走 `references/SUBTITLE-SYNC.md` 里的"无时间戳兜底方案"（客户端按标点/长度分句 + 按音频真实时长比例分配）。

**重新合成这一步顺手把语速也定下来**：默认语速通常偏慢，短视频场景建议整体提到 1.05-1.1 倍——但要先让用户听一段确认，不要凭经验值直接定案再全量重跑。这条和其余几条合成参数决策（换气声、专有名词读音、批量限流、增量重跑）详见 `references/SUBTITLE-SYNC.md` 的"合成参数调优"一节。

### S.2 / S.3 分组算法 + 运行时同步

完整规则、边界情况、和一份可以直接用的分组算法在 **`references/SUBTITLE-SYNC.md`**——里面记录了这几条踩过的坑，务必照做，不要凭直觉重新发明：

- 英文单词 / 文件名 / 代码路径（`SKILL.md`、`scripts/foo.py`）绝对不能被从中间切开
- 中文词语也尽量不要切开（比如"外部工具"不能变成"外部工"+"具"）——做法是快到字数上限时往后多看几个字，找到最近的标点再切，而不是数到数字就切
- 断句处的逗号/顿号是"因为要分屏所以断在这"的人为产物，不是真实语义停顿，前后都不要保留；只有真正的句末标点（。！？）才保留
- 播放同步用 `requestAnimationFrame` 轮询播放位置，不要用 `timeupdate` 事件（触发频率在部分浏览器下明显不够，字幕切换会看起来卡顿滞后）

`scripts/chunk-subtitle.mjs` 是这套分组算法的现成实现（纯函数，无依赖，$1 复制即可用）。

## Phase A · 讲师头像小窗口（可选）

### A.0 在画面生成前确定数字人布局

数字人不是成片阶段临时盖在 PPT 上的浮层。选择视觉风格时，必须紧接着增加一次数字人布局选择，并把结果保存为作品级配置：

- 是否使用数字人；
- 人物布局使用可视化画面卡片选择，不要求用户理解像素：右上角小窗、右下角小窗、右侧约三分之一讲师区；
- 右侧三分之一讲师区应为正文真正让出版面，而不是在成片阶段覆盖在 PPT 上；
- 用户确认布局后，才进入章节画面生成。

只要启用数字人，每一页 PPT 都必须从生成阶段开始为人物区域让位，文字、图表、数字、按钮和视觉焦点不得进入预留区域。对于已经生成完成的旧作品，修改数字人位置或大小后必须明确提示并重新生成或逐页重排全部章节，不能只在最后叠加视频。

工作台的数字人素材上传必须显示“上传中、上传成功、上传失败”状态；大文件限制也必须在界面明确反馈，不能让失败表现为点击后无反应。

完整方法论 + 真实踩过的坑在 **`references/AVATAR-PIPELINE.md`**，务必先读一遍。四个最关键、最反直觉的点：
1. **布局默认右侧竖屏大窗口 + 正文右侧留白**（不是右下角小圆窗）——见该文档第 0 节。
2. **切片不能跨乒乓翻转点**，否则动作会"弹"一下，而逐帧 seam 检查看不出来——第 2 节。
3. **offset 用"跟模型输出自带音频做交叉相关"求真实位置**，不要按时长比例估算——第 4 节。
4. **运行时把视频从动到音频时钟**（每帧 `offset+audio.currentTime`），不要"每步 seek"（会每句闪）、也不要"只段头 seek 段内自由播"（口型会越来越飘）——第 5 节。

开始前先问用户（如果对话里还没给出）：
1. 一段本人出镜的视频素材（哪怕只有几十秒——会用乒乓循环拉长）；竖屏素材直接适配右侧竖窗口
2. 用哪个对口型模型/API（本项目用的是本地 HeyGem，端口 7861，位置/启动见 `F:\Projects\MACHINE-INDEX.md` 服务表）
3. 布局细节（右窗口大小、正文右留白宽度）——这些是"会按项目调整的新要求"，确认一次

这一步涉及真实的模型调用（分钟级耗时、可能有 API 成本），跑之前把方案过一遍给用户看一眼再执行，不要静默跑完几十分钟才汇报。`scripts/align-avatar-offsets.py` 是第 4 步交叉相关对齐的现成实现（需 numpy+scipy）。

## Phase M · 手机字号体检

短视频几乎全是小窗口/手机观看，网页开发阶段在大屏上看着舒服的字号，缩小到手机宽度经常读不清。这是**几乎每个项目都会踩、而且会反复被用户指出**的问题，所以做完 Phase S/A 之后要主动过一遍，不用等用户抱怨。

完整检查清单在 **`references/MOBILE-SIZING.md`**，核心就一条铁律：

> **字看不清时，答案是"重新排版"（减少同屏元素、拉大间距），不是把 font-size 加两三个像素。**

跑完排版调整后，用 **`scripts/subtitle-overlap-sweep.js`**（贴到浏览器 devtools 或 `preview_eval` 里执行）过一遍全部 step，确认没有场景内容的底部被字幕条压住——这是加大字幕/加大正文字号之后最容易复发的次生问题，必须每次改完字幕条尺寸或大改某个 chapter 排版后都重新跑一遍。
