# 手机字号体检

短视频几乎全是小屏观看（手机竖屏、小窗口），网页开发阶段在电脑大屏上看着舒服的排版，缩小到手机宽度经常读不清。这是几乎每个项目都会反复被用户指出的问题，做完内容和动效之后要主动过一遍，不要等用户抱怨。

## 核心铁律

> **字看不清时，答案是"重新排版"，不是把 `font-size` 加两三个像素。**

反复出现的错误模式：反馈"字太小看不清"→ 把某个 class 的 font-size 从 13px 调到 15px → 用户还是觉得小，因为问题从来不是字号本身，而是"为了塞下所有信息，被迫用小字号"这个排版决策。正确做法是先问"这块内容如果只保留一半元素，够不够表达核心信息"，砍元素、拉大剩余元素，而不是全部保留、集体缩小一点点。

## 检查清单

过一遍每一个 chapter 的每一个 step，逐条检查：

- [ ] **一行里超过 3 个 badge/chip 挤在一起？** → 减少同屏数量，或让它们换行且加大间距/字号，而不是保持一行只是字号小一点。
- [ ] **任何手写 `font-size` 值 < 18px？** 核心信息（会被大声读出来/承载实际结论的文字）不允许小于 18px。装饰性/次要标签（比如页码、进度指示）可以例外，但要明确知道自己在例外。
- [ ] **全局共享的小字号 primitive class**（比如"小标签""kicker"一类几乎每屏都用的组件）有没有掉进这个坑？这类 class 一旦偏小，是全片性的问题，改一处能修全部场景——优先检查这些，性价比最高。
- [ ] **图形密集但文字也密集的场景**（比如网络节点图配一堆小标签）？优先做法是**压缩图形本身的留白、让图形更紧凑**，从而腾出空间给文字用更大的字号，而不是把图形和文字一起等比缩小——等比缩小会把字号也缩小，等于没解决问题。如果一定要缩小整个复合图形，缩小完之后要单独检查文字部分是否又变得不达标，不能假设"整体缩小了所以没问题"。
- [ ] **场景内容是否贴近画面底部？** 见下一节"字幕安全区"，这是加大字幕/加大正文之后最容易复发的次生问题。

## 什么是"真正的重新排版"（而不是敷衍的加大）

判断标准：改完之后把截图缩小到手机宽度（375px 左右）看一眼，核心结论是不是一眼可读——不用眯眼看、不用凭上下文猜内容。如果还要眯眼看，说明只是敷衍地加大了几个像素，没有真正解决拥挤问题。

一个真实的反例→正例：某场景是一个 SVG 小图 + 5 个 badge 挤在一行、8px 间距、16px 字号，旁边配一行说明文字。

- 敷衍加大：字号 16→18px，间距 8→10px——视觉密度几乎没变，用户会继续觉得小。
- 真正重排：badge 字号提到接近正文字号（比如 22-24px）、padding 明显加大、允许 5 个 badge 自然换行成 2 行而不是硬塞一行、说明文字从"图注级别"提升为"结论句级别"的字号和字重。

## 字幕安全区（Subtitle Safe Zone）—— 一个会反复复发的坑

现象：字幕做大、做醒目之后（比如从小字幕条改成大字号滚动提示条），之前从没出过问题的场景会突然"画面内容被字幕条挡住/紧贴"。这不是字幕组件的 bug，是场景内容原本就没有为"底部会有一条不透明遮罩"这件事预留空间。

常见根因（按出现频率排序，遇到新场景先怀疑这两类）：

1. **共享的 CSS class 里混进了"只有某一个 step 需要"的高度/定位规则**，被同一个 class 的其它 step 意外继承，把不相关内容拖到很低的位置。典型标志：某个 class 上有 `min-height` 或 `flex: 1` 撑满剩余空间，且这个 class 被多个 step 复用。修法：把这类规则限定到真正需要它的那个/那些 step 专属的 selector 上，不要留在共享的基础 class 里。
2. **绝对定位的图形节点/标签，坐标本身就超出了它声明的容器高度**（比如一个 `height: 600px` 的图表容器里，某个标签写着 `top: 640px`）。平时因为下面没有别的东西挡着看不出问题，字幕条一变高就露馅了。修法是**收紧图形内部节点的相对间距，让内容真正落在声明的高度以内**（把整体几何往中心压缩、缩短节点之间的连线距离），而不是对整个图形做等比缩小——等比缩小会连文字一起缩小，违反上面的核心铁律。只有先尝试收紧间距、真的收不下的极端情况，才考虑等比缩小作为最后手段，且缩小完必须单独检查文字有没有跟着变得不达标。

## 检测脚本

不用一屏一屏肉眼翻查。`../scripts/subtitle-overlap-sweep.js` 会自动过一遍指定范围内的所有 step，报出"内容最低点超出字幕条顶部多少像素"——正数表示有重叠需要修，个位数像素通常是动画过程中的采样噪声可以忽略，两位数以上必须修。

**什么时候必须跑一遍**：
- 改了字幕组件的高度/字号/padding 之后（全量跑一遍，这是最容易批量引入重叠的改动）
- 新增/大改任何一个 chapter 排版之后（至少跑该 chapter 对应的 step 范围）
- 交付前的最终检查（全量跑一遍）
