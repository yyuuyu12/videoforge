# 精确字幕同步

目标：字幕跟着音频真实播放位置切换，每次只显示一小段（~10 字），不是把整段口播文案糊在屏幕上等观众自己读完。

## 为什么不能用字数估算计时

估算法（"中文每字约 250ms"之类）在短句上误差还能接受，句子一长就会明显累积漂移，字幕会看起来"慢半拍"。只要 TTS 供应商能返回时间戳，就应该用真实数据。**没有时间戳数据的步骤才退化到估算**，且退化路径也要遵守本文档的分组规则（只是计时基准从"真实时间戳"换成"按字数比例分配到音频真实长度"）——不要因为没有精确时间戳就整段展示。

## 拿到时间戳：MiniMax 参考实现

`t2a_v2` 请求体加：

```json
{ "subtitle_enable": true, "subtitle_type": "word" }
```

支持的模型：speech-2.8-hd / speech-2.8-turbo / speech-2.6-hd / speech-2.6-turbo / speech-02-hd / speech-02-turbo / speech-01-hd / speech-01-turbo（写这份文档时的信息，用之前自己确认一下目前用的模型是否在列）。

返回体多一个字段 `data.subtitle_file`——24 小时有效的下载链接，内容是 JSON，逐字/逐句时间戳（毫秒）。**具体 JSON 里每个词条的字段名，第三方文档不一定写全**，务必先真实下载一条数据看一眼实际结构，再写解析代码：

```js
const resp = await fetch(subtitleUrl);
const words = await resp.json();
console.log(JSON.stringify(words).slice(0, 500)); // 先看长什么样
```

常见候选字段名（写解析器时做防御式兼容，遇到没见过的结构直接报错并打印 keys，不要静默吞掉）：
- 文本：`text` / `word` / `content`
- 起始时间：`time_begin` / `begin_time` / `start_time` / `start`
- 结束时间：`time_end` / `end_time` / `stop_time` / `end`

换了别的 TTS 供应商，先确认它是否有类似能力（多数主流 TTS 都有，常需要显式传参开启），字段名按它自己的文档来。

## 顺手一起做：合成参数调优（重新合成的时候就该定下来，别之后单独再返工）

既然 Phase S 要重新跑一遍合成，下面这几条参数决策放在这一步一起做，不要留到最后才发现语速不对又要再合成一遍：

- **语速**：不要用供应商默认值（通常是 1.0）就定稿。默认语速配合"讲解型"文案通常偏慢，短视频场景建议整体提到 **1.05-1.15 倍**（本项目用户既定偏好 **1.12**），但一定要让用户实际听一段确认后再批量跑——语速偏好因人而异，不能凭经验值直接定案就全量重新合成。
- **情绪 emotion**：MiniMax `t2a_v2` 的 `voice_setting.emotion`（`happy`/`sad`/`angry`/`fearful`/`disgusted`/`surprised`/`calm`）**对克隆音色也生效**（已实测，不只是内置预设音色）。默认平淡的口播加一个贴合内容基调的情绪会明显更有感染力（本项目文案是"呼吁/批判"基调，用户既定偏好 **angry**）。同样要 A/B 试听后再定。
- **A/B 试听不要靠反复全量重跑**：改语速/情绪这类参数时，先用 1 句真实文案合成 3-5 个候选（不同语速×情绪组合），把 mp3 用 data-URI 塞进一个 `<audio>` 对比页面（Artifact）让用户选定，再全量重合成。全量重合成很贵（下游字幕、对口型全要跟着重跑），别拿它当试听手段。
- **换气声/呼吸口**：如果 TTS 输出带有明显换气声导致节奏拖沓，优先看供应商是否有"去除呼吸音"一类的参数；没有的话，提高语速本身也会间接压缩换气声的相对时长占比，是个还算有效的副作用。
- **专有名词读音**：技术类文案里的英文缩写/文件名（比如 `SKILL.md`）容易被 TTS 按错误的方式拆读断句。发现读错，去改**文案本身**做"发音友好化"处理，不要指望调 TTS 参数解决——大多数供应商的参数没有这么细粒度的控制。
- **批量合成要做限流**：多数 TTS API 有 RPM（每分钟请求数）限制，几十条无间隔并发提交会触发限流报错。批量脚本里加一个固定的请求间隔（比如 1.5 秒）比事后重试更省心。
- **要支持增量重跑**：合成脚本要能"已存在的文件跳过，加一个 `--force` 才强制重新生成"——语速/文案调整后经常需要重新合成，中途失败/只改了几条文案时没有增量能力会很痛苦。

## 分组算法：把逐字时间戳变成"一屏一句"

逐字时间戳太碎，不能直接当字幕用。分组规则：

- **正常中文目标 8 字，硬上限约 10 字**：每次只显示一行短语；只有不可拆的英文、数字或代码 token 才允许临时超过，绝不能为了凑完整句扩成两行。
- **字幕只显示当前段**：下一段开始时上一段立即隐藏，不保留历史字幕，不同时堆叠前后两段。

- **绝不切开一个 token**：英文单词、文件名、代码路径（`SKILL.md`、`scripts/foo.py`、`brand-guidelines`）当作一个不可分割的整体处理。判断方法：把文本先按 `[A-Za-z0-9_./-]+` 切出"原子"，剩下的每个字符（包括中文字、标点、空格）各自是一个原子，之后所有分组逻辑都在"原子"这个粒度上操作，而不是原始字符粒度——这样天然不会切到英文单词内部。
- **优先在真实停顿处断句**：目标长度 8 字，中文硬上限 10 字；10 字内遇到标点立即断句，没有标点则在第 10 字强制切换。只有不可拆的英文、数字或代码 token 可以单独超过 10 字。
- **断句处的逗号/顿号不保留，句末的句号/问号/叹号保留**：逗号出现在分屏点上，是因为"要在这里分屏"这个展示决策造成的，不是原文真的在这里停顿了很久——保留会让字幕看起来没说完就被截断了一样。真正的句末标点则相反，代表一个完整意思结束，保留是对的。这条规则要两个方向都做：前一段末尾不能有逗号收尾，后一段开头也不能有逗号起头（如果因为字数上限刚好卡在标点前一个字，下一段开头就会遗留这个标点，必须显式跳过）。
- **一个超长 token 单独起一段**：如果一个不可分割的英文/代码 token 加上前面已经攒的文字会导致这一屏严重超长（比如 `brand-guidelines` 17 个字符，前面还攒了十几个字），要先把前面攒的内容单独收一段，让这个 token 从下一段的开头重新开始，而不是硬凑成一段又长又不好读的字幕。

一份可以直接抄的实现在 `../scripts/chunk-subtitle.mjs`（纯函数，零依赖）。

## 运行时同步

- 用 `requestAnimationFrame` 轮询音频的 `currentTime`，每帧判断"当前该显示哪一段"（找最后一个 `startMs <= 当前播放时间` 的段）。**不要用 `audio` 的 `timeupdate` 事件**——部分浏览器下这个事件每秒只触发几次，字幕切换会看起来滞后卡顿。
- 判断逻辑只依赖每段的**起始时间**，不依赖结束时间——下一段开始了，上一段自然就该消失，这样即使某个字的时间戳误差稍大也不影响"该显示哪一段"的判断。
- 没有真实时间戳的步骤（兜底路径）：拿音频元素的真实 `duration`（不是字数估算出的时长），按每段字数占全文字数的比例分配起始时间——这样即使没有逐字数据，起码是按真实音频长度等比例分配，比纯字数估算准。

## UI 呈现

- 大字号、单行、居中，不要允许自动换行到第二行（分组算法本身已经保证每段够短），否则一旦某段异常长就会占用大片画面。
- **字幕样式先给一个"模板化"的（半透明黑底渐变 scrim + 上浮动画）不一定是对的**——本项目用户明确否掉了模板样式，两轮反馈后定稿为**纯黑字、无底框、无背景、无动画**（`color:#000`，不加 `background`/`border-radius`/`padding`/入场动画）。这在浅色主题（如 newsroom 米白底）上清晰可读。经验：字幕样式倾向"越简越好"，不要默认套一个带底框/动效的模板；而且纯深色字只在浅底主题上安全，深底主题上要反过来用浅色字。开工时按项目主题底色决定字色，样式繁简先给最简的让用户加，而不是给最花的让用户砍。
- （若采用带动画的样式）每段切换时 retrigger 一个轻微的进出场动画（比如 200-300ms 的上浮淡入），让"切换"这件事本身有反馈感，而不是文字突然跳变。
- 没有任何字幕数据可用的最终兜底（比如画面验收阶段还没生成音频）：也必须先按同一规则切成约 8~10 字的短段，每次只显示一段单行字幕并定时轮换。禁止直接展示整段口播，禁止两行兜底。
